Lectures 5 and 6 built differential privacy for statisticians: private means, quantiles, medians, subgroup gaps, and the local/smooth-sensitivity machinery that made one audited number safe to release. This lecture keeps the audit habit — proposal, diagnosis, repair — but changes what gets released. Instead of a scalar, the release is a model: a weight vector \(\hat w\) that a downstream user will multiply against new feature vectors it has never seen. The question is the same one as always (how much can a single row move the release?), but the answer now depends on geometry — the shape of the feature cloud — in a way a scalar mean never had to.

Running example. We use the Communities and Crime benchmark (sml gate-selection below): each row describes a community, the feature columns are demographic and economic covariates, and the label is a crime-rate statistic. In a fully private pipeline the bounds on \(\|x_i\|_2\) and \(|y_i|\) would themselves have to be privately estimated — exactly the clipping problem Lecture 5 spent three sections repairing. Here those bounds are fixed so this lecture’s attention stays on regression.

Public contract (row-DP): \(\|x_i\|_2\le 1\), \(|y_i|\le 1\), adjacency is fixed-size row replacement, \(\delta=1/n^2\). Nothing about a row is public by default — not the features, not the label. That is the harder element contract, and getting it right is most of this lecture.

Where OLS lives inside the contract. Ordinary least squares solves \(\hat w=(X^\top X)^{-1}X^\top y\); ridge solves the stabilized version \(\hat w_\lambda=(X^\top X+\lambda I)^{-1}X^\top y\). Both formulas route through exactly two data-dependent objects: \(A=X^\top X\) and \(b=X^\top y\). \(A\) is pure feature geometry — which directions in feature space are backed by many rows and which are nearly empty. \(b\) is the feature–label relationship — the direction the labels actually pull the fit toward. Privacy noise can land on \(A\), on \(b\), or on \(\hat w\) itself, and where it lands turns out to matter more than the total amount of noise. That is the question the rest of this post answers.

Interactive geometry: what does one row buy?

Drop to two dimensions so a single row is visible. The picture below is the same OLS line-fitting problem as the baseline above, just small enough to reason through directly: one feature, one label, a handful of points.

Question to answer before dragging: if you move one point, does the fitted line move a lot or a little — and does the answer depend on where the point sits, or only on how far you move it?

Two ideas govern the answer, and they will reappear under every mechanism below:

  • Leverage is a point’s voice in the regression problem — how much the fitted line’s direction depends on that one point’s feature location. A point near the center of the feature cloud has almost no leverage: the other points already agree on where the line should go in that neighborhood, so moving the center point barely rotates anything. A point far out along a direction where the rest of the data provides little support has high leverage: it is close to the only evidence the fit has about that direction, so its label can swing the whole line.
  • Residual is the other half of influence. A high-leverage point only matters when its label disagrees with what the rest of the data would already predict there. A high-leverage point that happens to sit right on the trend moves nothing when nudged; the same point with a surprising label can drag the fit substantially.

Drag the highlighted point below and compare the refit line to the frozen leave-one-out reference (the fit computed without that point). The gap between the two lines is the point’s influence — the same replacement-adjacency idea Lecture 5 used for the median and Lecture 6 used for subgroup gaps, now applied to a fitted model instead of a fitted number.

Diagnosis. Two facts fall out of the drag experiment, and both are privacy-relevant, not just statistical trivia:

  1. Influence is not uniform across rows. A privacy mechanism that adds the same noise to every row (or the same noise regardless of position) is implicitly assuming every row has the same leverage. The picture above shows that assumption is false for essentially any real feature cloud.
  2. Worst-case leverage, not typical leverage, sets the privacy bound. A typical row in the dense middle of the cloud can move a fraction of a degree; a single point placed at the edge of the feature ball can move the line by an amount limited only by the label bound. Differential privacy has to protect against the second row, even though almost every row looks like the first. This is the same worst-case-versus-typical-case tension from Lectures 5–6 (a substitution that barely moves the empirical mean versus one engineered to move it a lot), now restated in feature-space geometry instead of value space.

Before adding any noise, one more distinction has to be nailed down: which part of a row is private? That question splits the rest of the lecture in two.

Label-DP versus element-DP. In label-DP, the feature vectors \(X\) are treated as public and only the labels \(y\) are private; a neighboring dataset keeps every feature vector fixed and replaces one label. This is a useful pedagogical stepping stone — in the 1D picture above, a label-only replacement can only move a point vertically. In element-DP (the row-DP contract this lecture actually targets), the whole record \((x,y)\) may change: a neighboring dataset can replace both a feature vector and its label. Element-DP is strictly harder, because now the adversary also controls where in feature space the changed row lands — and the leverage argument above says that location, not just the label value, controls how much the fit can move. Every mechanism below is evaluated primarily under the harder element-DP contract; label-DP appears once, next, as a diagnostic warm-up because it isolates the “move only vertically” case before the full “move anywhere” case.

Proposal 1: input perturbation (label-DP diagnostic)

Proposal. The most direct private-regression strategy needs no new theory: add enough noise to each row so that releasing the noisy rows is private, then run ordinary OLS on the noisy data as post-processing. Whatever comes out is private because it is a deterministic function of an already-private release. This works for any mechanism downstream, not just OLS, which is exactly why it is tempting.

Why it is wasteful. Input perturbation is a row-DP mechanism in disguise of a local-DP one: because the noise must make each individual row safe to release on its own, the noise scale cannot shrink as \(n\) grows — every one of the \(n\) rows pays for its own privacy, rather than the \(n\) rows jointly paying once for a shared statistic. Concretely, under label-DP a single label has sensitivity \(2\) (it can move by the full \([-1,1]\) range in either direction), so the noise added to each label must be calibrated to sensitivity \(2\). But the quantity the learner actually needs — the sufficient statistic \(b=X^\top y\) — only has per-row sensitivity \(2/n\): one row’s label can move \(b\) by at most \(2\cdot\max_i\|x_i\|_2 \le 2\) in absolute terms, but that single row’s share of a sum over \(n\) rows shrinks the marginal contribution as \(n\) grows in the way input perturbation never gets to exploit. The next two figures make this diagnostic concrete: first the error distribution of the input-perturbed release, then the effect on the fitted line’s slope in a witness where noise-versus-signal is visible by eye.

Diagnosis (label-DP input perturbation, diagnostic foil — not the repair). The error distribution above is centered but wide, and the low-slope witness shows why: once every label carries independent sensitivity-\(2\) noise, the signal in \(X^\top y\) (which depends on how strongly features and labels actually correlate) has to compete with noise that was calibrated to the worst-case single-label range rather than to the statistic’s true sensitivity. On a design where the true slope is small to begin with, the noisy slope is barely distinguishable from zero. Input perturbation is always available — it needs no assumptions about the learner beyond the public bounds — but it is never the mechanism of choice once a repair that targets the sufficient statistic directly is on the table. That repair is the second half of this lecture. First, one more foil: what if we skip the noisy-input step and add noise to \(\hat w\) at the very end instead?

Proposal 2: output perturbation (diagnostic foil — fails)

Proposal. Run OLS exactly as usual on the real data, then add noise to the fitted weight vector \(\hat w\) before releasing it. This is the output-perturbation template that works cleanly for many scalar statistics (mean, count) — calibrate the noise to the statistic’s global sensitivity and add it once at the end.

Question to answer before the figure: does \(\hat w=(X^\top X)^{-1}X^\top y\) have a global sensitivity at all — a single number, independent of the dataset, that bounds how much one row replacement can move \(\hat w\)?

Why it fails. It does not, in general. \(\hat w\) passes through \((X^\top X)^{-1}\), and a single high-leverage row is exactly a row that can push \(X^\top X\) toward a nearly singular matrix along the weakly supported direction identified above. As the smallest eigenvalue of \(X^\top X\) approaches zero, \((X^\top X)^{-1}\) blows up, and with it the amount that one replacement can move \(\hat w\) — unboundedly, in the worst case, for ordinary (unregularized) linear regression. A mechanism cannot calibrate finite noise to an unbounded sensitivity. The figure below is therefore a diagnostic, not a row-DP claim: it shows how the effective sensitivity of \(\hat w\) scales with the conditioning of \(X^\top X\) on an engineered near-singular witness, without asserting any formal \((\epsilon,\delta)\) bound for the underlying (non-private) \(\hat w\) release.

make_output_perturbation_conditioning_figure(geometry, witness="nearly_singular")

Diagnosis (output perturbation, diagnostic foil — not a private mechanism as shown). The conditioning witness above is engineered so the feature cloud is nearly degenerate along one direction — the same low-support direction the leverage discussion flagged earlier. As that direction’s eigenvalue shrinks, the diagnostic sensitivity of \(\hat w\) grows without a visible ceiling. This is the regression analogue of a denominator that can approach zero: any private statistic that requires dividing by (or inverting) a quantity the data itself controls needs a worst-case floor on that quantity before it can be calibrated, and ordinary OLS provides none. Two routes repair this. One is to bound \(\hat w\) itself by regularizing the solve (ridge, coming up). The other — the one this lecture takes first — is to stop perturbing the model at the end and instead perturb the sufficient statistics the model is built from, where the sensitivity is easy to state honestly.

Repair: noisy sufficient statistics, then ridge

Proposal. Squared-loss linear regression has a natural pair of sufficient statistics: \(A=X^\top X\in\mathbb R^{d\times d}\) and \(b=X^\top y\in\mathbb R^d\). If \((A,b)\) are released privately, then any function of them — including \(\hat w=A^{-1}b\) or \(\hat w_\lambda=(A+\lambda I)^{-1}b\) — is free post-processing. Unlike \(\hat w\), both \(A\) and \(b\) have clean, dataset-independent sensitivity bounds under the row-DP contract: \(b=X^\top y\) is a vector-valued linear query (one row changes it by a bounded amount because \(\|x_i\|_2\le1\) and \(|y_i|\le1\)), and \(A=X^\top X\) is a sum of bounded rank-one outer products \(x_ix_i^\top\), so one row changes \(A\) by a bounded amount in spectral norm too. Adding Gaussian noise to both is therefore a valid private release — this is the Analyze Gauss template: privatize \((A,b)\), solve afterward.

So sufficient-statistics perturbation (SSP) is privacy-valid the moment output perturbation was not. But privacy validity was never the hard part of output perturbation — utility was, and noising \(A\) directly reopens exactly the same wound from the other side. The figure below perturbs \(X^\top X\) on the same near-singular witness used for the output-perturbation diagnostic. Watch what happens to the weakly supported direction: additive noise on a matrix that already has a near-zero eigenvalue does not just blur that direction, it can flip its sign, making the noisy matrix indefinite and \(\hat w = \tilde A^{-1}\tilde b\) either wildly inaccurate or literally uninvertible.

Ridge is the natural repair, not a generic regularizer. Ridge solves \((A+\lambda I)w=b\) instead of \(Aw=b\). Adding \(\lambda I\) lifts every eigenvalue of \(A\) by \(\lambda\) before inversion — it is exactly the stabilization the noisy near-zero eigenvalue needs, applied for the same structural reason a denominator floor was needed above. Reaching for ridge here is not a stylistic preference; it is the direct fix for the failure mode the previous two figures diagnosed.

AdaSSP turns this into a full algorithm: perturb \(A\) and \(b\) with Gaussian noise calibrated to their sensitivities, privately estimate how small the true minimum eigenvalue of \(A\) is likely to be (using a further slice of the privacy budget), and set the ridge weight \(\lambda\) large enough to dominate the estimated noise-to-signal ratio in the weakest direction — then solve \(\hat w=(\tilde A+\lambda I)^{-1}\tilde b\). The three steps (noise \(A\), noise \(b\), estimate \(\lambda_{\min}\) privately) compose under the usual Gaussian-mechanism accounting; the final solve is post-processing.

Gaussian sketching is a second way to privatize the Gram side. Instead of adding coordinate-wise noise directly to \(A\), it draws a random Gaussian sketch of the rows and forms a noisy Gram matrix from the sketch, mixing in a residual noise floor calibrated the same way AdaSSP calibrates its ridge weight. Geometrically this is still “randomize the covariance-like object,” just through projection instead of additive coordinate noise, and it inherits the same leverage-driven difficulty: high-leverage rows still dominate the sketch, so a ridge-style stabilization step is still needed afterward. The right-hand side \(b\) is privatized the same way as in AdaSSP.

Provenance caveat. The Gram-sketch repair used here is a local teaching implementation of a one-step Hessian-mixing idea. Fixed-seed tests guard the current wiring, but this lecture does not claim a fully audited upstream port.

Diagnosis. Compare this panel to the output-perturbation conditioning figure above: same witness, same weak direction, but now the noise lands on \(A=X^\top X\) instead of on \(\hat w\). The noisy ellipse visibly loses its shape precisely along the low-eigenvalue axis — the direction with the fewest rows backing it is the direction where additive noise is proportionally largest relative to the true signal. This is the utility question SSP raises that output perturbation never even got to ask, because output perturbation had already failed on privacy validity. AdaSSP’s answer is to measure how bad this direction is (privately) and size the ridge term to it; a fixed, untuned \(\lambda\) would either under-stabilize on badly conditioned data or over-shrink on well-conditioned data. Diagnosis. Element input perturbation and output perturbation are shown here only as diagnostics — they are not being claimed as valid row-DP releases at this \(\epsilon\), for the reasons diagnosed above (wasteful per-row noise; unbounded worst-case sensitivity). AdaSSP and the Gram-sketch repair are the two mechanisms in this comparison that are valid row-DP releases under the stated contract, and they consistently place noise at the sufficient-statistics stage rather than the row stage or the final-weight stage. Neither is free: both still pay for the geometry of the problem, in particular for how small the minimum eigenvalue of \(X^\top X\) is and how much of the feature space is only weakly supported by data. What algorithm design buys is where that price is paid — at the statistic, where sensitivity is small and well understood, rather than at every row or at an unbounded final quantity.

Closing: privacy is a worst-case statement about influence

Every method in this lecture answers the same question — how much can one row change the release — but at a different stage of the pipeline: at the row (input perturbation), at the final model (output perturbation, which fails), or at the sufficient statistics (SSP, AdaSSP, Gram-sketch). The quantity that decided the outcome at every stage was the same one the drag interactive introduced at the start: leverage, the geometric fact that some feature directions are backed by almost no data and therefore let one row move the fit a great deal.

That quantity is not unique to privacy. Robust statistics asks the identical question — how much can one outlier move an estimator — and answers it with the same leverage/residual decomposition used here. Generalization asks a cousin question — how much can one training point move a model’s predictions on new data — and stability-based generalization bounds are built from the same influence quantities. What makes the private version distinct is the quantifier: privacy insists on a bound that holds for the single worst possible row, chosen adversarially, while ordinary statistical reasoning is usually content with a bound that holds for a typical row, or on average across rows. A method can be statistically robust in the typical case (the low-slope witness looked fine on ordinary data) and still fail a worst-case privacy audit (the near-singular witness broke output perturbation completely) — the recurring worst-case-versus- typical-case contrast from Lectures 5–6, now stated in feature-space geometry instead of value space.

The private algorithm designer’s job, across every mechanism surveyed here, was to choose where in the computation to spend the noise so that it damages the least amount of useful signal while still bounding every individual row’s worst-case influence — and, when the raw sufficient statistics turn out to be too fragile after noise (a near-singular \(\tilde A\)), to repair the solve itself with ridge rather than pretend the fragility away.

Bridge to Lecture 8. Squared loss let this lecture collapse an entire model down to two noisy sufficient statistics, \(\tilde A\) and \(\tilde b\), with the model itself recovered as post-processing. The next lecture asks what happens when the loss is no longer squared: a linear-query classifier whose weights are literally \(w=\frac{1}{n}\sum_i y_i x_i\) — a single vector-valued linear query with a much simpler sufficient statistic, but a discontinuous 0/1 loss that reintroduces the leverage problem in a new form.
